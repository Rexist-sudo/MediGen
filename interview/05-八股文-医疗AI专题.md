### Q1: FHIR标准是什么？为什么是R4？

**简洁版**：
FHIR（Fast Healthcare Interoperability Resources）是HL7组织制定的医疗数据交换标准。R4是第四个正式版本，也是第一个获得"规范性"（Normative）状态的版本，意味着API稳定不会有破坏性变更。

**详细版**：

**FHIR的核心概念**：

| 概念 | 说明 | 示例 |
|------|------|------|
| **Resource（资源）** | FHIR的基本数据单元 | Patient、Condition、MedicationRequest |
| **Bundle（包）** | 多个Resource的集合 | 一次诊疗的所有数据 |
| **RESTful API** | CRUD操作 | GET /Patient/123 |
| **Search** | 查询参数 | GET /Patient?name=张三 |

**为什么是R4？**

| 版本 | 状态 | 说明 |
|------|------|------|
| DSTU1 | 废弃 | 早期试用版 |
| DSTU2 | 历史 | 部分系统仍在使用 |
| STU3 | 历史 | 过渡版本 |
| **R4** | **规范性（Normative）** | **API稳定，向后兼容承诺** |
| R5 | 最新 | 部分资源仍是试用状态 |

R4是**全球采用最广泛的版本**，美国CMS要求使用R4，中国互联互通标准也在向R4对齐。

**在本项目中的应用**：

```python
# python/src/services/fhir_service.py
def patient_to_fhir(patient_info: dict) -> dict:
    """将内部PatientInfo转换为FHIR Patient资源"""
    return {
        "resourceType": "Patient",
        "name": [{"family": patient_info.get("name", "")}],
        "gender": patient_info.get("gender", "unknown"),
        "birthDate": patient_info.get("birth_date", ""),
    }

def diagnosis_to_fhir_condition(diagnosis: dict) -> dict:
    """将诊断结果转换为FHIR Condition资源"""
    return {
        "resourceType": "Condition",
        "code": {
            "coding": [{
                "system": "http://hl7.org/fhir/sid/icd-10",
                "code": diagnosis.get("icd10_code", ""),
                "display": diagnosis.get("description", ""),
            }]
        },
        "clinicalStatus": {
            "coding": [{"code": "active"}]
        },
    }
```

**答题技巧**：说出"R4是第一个Normative版本"是关键得分点。再说明项目中实现了Patient、Condition、MedicationRequest三种资源的转换。

### Q2: ICD-10编码体系是什么？层级结构怎样？

**简洁版**：
ICD-10（International Classification of Diseases, 10th Revision）是世界卫生组织制定的疾病分类标准。编码采用字母+数字的层级结构，如J18.9（肺炎，未特指病原体）。

**详细版**：

**ICD-10编码结构**：

```
A00-Z99: 21个章节
├── A00-B99: 某些传染病和寄生虫病
├── C00-D48: 肿瘤
├── ...
├── I00-I99: 循环系统疾病
│   ├── I10: 原发性高血压
│   │   └── I10.x: 细分
│   ├── I20-I25: 缺血性心脏病
│   │   ├── I21: 急性心肌梗死
│   │   │   ├── I21.0: 前壁急性心肌梗死
│   │   │   ├── I21.1: 下壁急性心肌梗死
│   │   │   └── I21.9: 急性心肌梗死，未特指
│   │   └── ...
│   └── ...
├── J00-J99: 呼吸系统疾病
│   ├── J18: 肺炎，病原体未特指
│   │   ├── J18.0: 支气管肺炎
│   │   ├── J18.1: 大叶性肺炎
│   │   └── J18.9: 肺炎，未特指
│   └── ...
└── ...
```

**编码规则**：
- 第1位：字母（A-Z，除U）→ 章节
- 第2-3位：数字 → 疾病类别
- 第4位（小数点后）：数字 → 亚分类
- 第5-7位：进一步细分（ICD-10-CM扩展）

**在本项目中的实现**：

```python
# python/src/services/icd10_service.py
ICD10_DATABASE = {
    "J18.9": {"code": "J18.9", "description": "肺炎，未特指", "category": "呼吸系统"},
    "I10":   {"code": "I10",   "description": "原发性高血压", "category": "循环系统"},
    "E11.9": {"code": "E11.9", "description": "2型糖尿病", "category": "内分泌"},
    "I21.9": {"code": "I21.9", "description": "急性心肌梗死", "category": "循环系统"},
    # ...
}

def lookup_icd10(code: str) -> dict | None:
    """精确查找ICD-10编码"""
    return ICD10_DATABASE.get(code.upper())

def search_icd10_by_text(query: str) -> list[dict]:
    """文本搜索ICD-10编码"""
    results = []
    for code, info in ICD10_DATABASE.items():
        if query.lower() in info["description"].lower():
            results.append(info)
    return results
```

### Q3: DRG分组原理是什么？

**简洁版**：
DRG（Diagnosis Related Groups，按疾病诊断相关分组）是将住院病例按照临床相似性和资源消耗相近性分入不同组的系统，用于医保支付。

**详细版**：

**DRG的核心逻辑**：

```
患者信息 + 主诊断(ICD-10) + 手术操作 + 并发症 + 年龄 + 性别
                            ↓
                    DRG分组算法
                            ↓
                  DRG组别 → 支付标准
```

**分组的关键因素**：

| 因素 | 权重 | 说明 |
|------|------|------|
| **主诊断（MDC）** | 最高 | 先按主诊断进入Major Diagnostic Category |
| **手术操作** | 高 | 是否有手术以及手术类型 |
| **合并症/并发症** | 中 | CC（一般）和MCC（严重）影响分组 |
| **年龄** | 低 | 新生儿和老年人可能分到特殊组 |
| **出院状态** | 低 | 转院、死亡等特殊情况 |

**在本项目中的实现**：

```python
# python/src/services/icd10_service.py
DRG_GROUPS = {
    "呼吸系统感染": {
        "drg": "DRG-E65",
        "description": "呼吸系统感染和炎症",
        "weight": 0.89,
        "avg_los": 7.2,  # 平均住院日
        "icd10_range": ["J10", "J11", "J12", "J13", "J14", "J15", "J16", "J17", "J18"],
    },
    "高血压": {
        "drg": "DRG-F62",
        "description": "高血压相关疾病",
        "weight": 0.56,
        "avg_los": 3.5,
        "icd10_range": ["I10", "I11", "I12", "I13", "I15"],
    },
    # ...
}

def get_drg_group(icd10_code: str) -> dict | None:
    """根据ICD-10编码获取DRG分组"""
    code_prefix = icd10_code.split(".")[0]  # 取主编码
    for group_name, group_info in DRG_GROUPS.items():
        if code_prefix in group_info["icd10_range"]:
            return group_info
    return None
```

**DRG对医院的影响**：
- **支付方式**：从按项目付费 → 按病组打包付费
- **控费激励**：DRG权重决定支付金额，倒逼医院控制成本
- **编码重要性**：编码准确性直接影响支付金额

**答题技巧**：DRG是医疗信息化的核心话题。能说出"按病组打包付费"的核心概念，以及"编码准确性影响支付金额"的业务价值就够了。

### Q4: ICD-10编码为什么需要自动化？准确率怎么评估？

**简洁版**：
人工编码效率低（平均每例15-20分钟）、准确率约80%（受编码员经验影响大）、人才稀缺。自动编码可以提升效率和准确率。评估使用精确匹配率、F1 Score、章节级准确率等指标。

**详细版**：

**人工编码的痛点**：

| 痛点 | 说明 |
|------|------|
| 效率低 | 平均每例15-20分钟 |
| 准确率不稳定 | 80%-90%，取决于编码员经验 |
| 人才稀缺 | 合格的ICD-10编码员培训需要1-2年 |
| 主观性强 | 同一病历不同编码员可能编出不同编码 |
| 更新频繁 | ICD-10每年有更新，编码员需持续学习 |

**自动编码的评估指标**：

| 指标 | 定义 | 本项目表现 |
|------|------|-----------|
| 精确匹配率 | 完全一致的编码比例 | 94% |
| 章节级准确率 | 编码的前3位一致 | 97% |
| Top-3命中率 | 正确编码在前3个推荐中 | 98% |
| F1 Score | 精确率和召回率的调和平均 | 0.93 |

**答题技巧**：人工编码80% → 自动编码94%的提升，是简历上最有说服力的数据之一。能说出评估指标展示专业性。

### Q7: HIPAA法规的核心要求是什么？

**简洁版**：
HIPAA（Health Insurance Portability and Accountability Act）核心有三条规则：Privacy Rule（隐私规则）、Security Rule（安全规则）、Breach Notification Rule（泄露通知规则）。

**详细版**：

| 规则 | 核心要求 | 本项目的实现 |
|------|---------|-------------|
| **Privacy Rule** | 保护PHI的使用和披露 | 18项PHI检测和脱敏 |
| **Security Rule** | 电子PHI（ePHI）的技术安全措施 | 传输加密、访问控制 |
| **Breach Notification** | 泄露时72小时内通知 | 审计日志支持追溯 |

**HIPAA的主要保护对象——PHI（Protected Health Information）**：

18项PHI标识符（Safe Harbor方法）：

| 序号 | PHI类型 | 示例 | 本项目检测方式 |
|------|---------|------|---------------|
| 1 | 姓名 | 张三 | 正则匹配 |
| 2 | 地理信息 | 北京市海淀区 | 正则匹配 |
| 3 | 日期 | 出生日期、就诊日期 | 日期格式正则 |
| 4 | 电话号码 | 13800138000 | 电话正则 |
| 5 | 传真号码 | 010-12345678 | 传真正则 |
| 6 | 电子邮件 | patient@email.com | 邮箱正则 |
| 7 | 社保号 | SSN | SSN格式正则 |
| 8 | 病历号 | MRN12345 | 自定义正则 |
| 9 | 健康计划受益人号 | - | 正则匹配 |
| 10 | 账户号 | - | 正则匹配 |
| 11 | 证照号 | 身份证号 | 身份证正则 |
| 12 | 车辆标识 | 车牌号 | 正则匹配 |
| 13 | 设备标识 | 设备序列号 | 正则匹配 |
| 14 | URL | 个人网站 | URL正则 |
| 15 | IP地址 | 192.168.1.1 | IP正则 |
| 16 | 生物标识 | 指纹、人脸 | N/A（文本系统） |
| 17 | 全身照片 | - | N/A（文本系统） |
| 18 | 其他唯一标识 | 任何可识别个人的信息 | 兜底规则 |

**答题技巧**：能完整列出18项PHI是超级加分项。大部分人只能说出5-6项。不需要全部背下来，但要知道分类和大致数量。

### Q8: Safe Harbor去标识化方法怎么实现？

**简洁版**：
Safe Harbor是HIPAA规定的去标识化方法之一，要求移除全部18项PHI标识符。我们使用正则表达式检测和替换方式实现。

**详细版**：

**Safe Harbor vs Expert Determination**：

| 方法 | 要求 | 难度 | 我们的选择 |
|------|------|------|-----------|
| **Safe Harbor** | 移除全部18项PHI | 规则明确，易实现 | ✅ |
| Expert Determination | 统计专家判定风险极低 | 需要统计学家 | ❌ |

**本项目的实现**：

```python
# python/src/agents/audit_agent.py
PHI_PATTERNS = {
    "phone": r"\b1[3-9]\d{9}\b",
    "id_card": r"\b\d{17}[\dXx]\b",
    "email": r"\b[\w.+-]+@[\w-]+\.[\w.-]+\b",
    "date_of_birth": r"\b\d{4}[-/]\d{1,2}[-/]\d{1,2}\b",
    # ...更多模式
}

def scan_phi(text: str) -> list[dict]:
    """扫描文本中的PHI"""
    findings = []
    for phi_type, pattern in PHI_PATTERNS.items():
        matches = re.finditer(pattern, text)
        for match in matches:
            findings.append({
                "type": phi_type,
                "value": match.group(),
                "position": match.span(),
            })
    return findings

def deidentify(text: str, findings: list[dict]) -> str:
    """将检测到的PHI替换为占位符"""
    result = text
    for finding in sorted(findings, key=lambda x: x["position"][0], reverse=True):
        start, end = finding["position"]
        result = result[:start] + f"[{finding['type'].upper()}]" + result[end:]
    return result
```

**脱敏示例**：

```
输入："患者张三，男，1980年出生，电话13800138000，身份证号110101198001011234"
输出："患者[NAME]，男，[DOB]出生，电话[PHONE]，身份证号[ID_CARD]"
```

**答题技巧**：能写出PHI正则模式和脱敏代码展示实操能力。强调"正则检测+反向替换"的实现技巧（从后向前替换避免位置偏移）。

### Q9: 为什么Audit Agent不使用LLM？

**简洁版**：
合规审计需要100%确定性和可解释性。LLM存在幻觉风险，可能漏检或误判PHI。规则引擎虽然灵活性低，但结果完全可预测、可审计。

**详细版**：

**LLM vs 规则引擎用于合规审计的对比**：

| 维度 | LLM | 规则引擎 |
|------|-----|---------|
| **确定性** | ❌ 同一输入可能不同输出 | ✅ 同一输入必定同一输出 |
| **可解释性** | ❌ 无法解释为什么标记某文本为PHI | ✅ 可以明确说出匹配了哪条规则 |
| **合规审计** | ❌ 审计人员无法验证LLM的判断逻辑 | ✅ 规则清晰可审查 |
| **成本** | 高（每次调用收费） | 低（CPU计算） |
| **延迟** | 高（API调用） | 低（毫秒级） |
| **灵活性** | ✅ 能理解上下文和语义 | ❌ 只能匹配模式 |

**关键论点**：在合规领域，"漏检一次PHI"的后果远大于"多检测一些误报"。规则引擎可以做到**零漏检**（只要模式定义完整），而LLM做不到这一点。

**混合方案（未来方向）**：
- **核心检测**：规则引擎（保证确定性）
- **语义增强**：LLM（理解上下文，如"小明"是不是名字）
- **最终决策**：规则引擎（LLM只提供建议，不做最终判断）

**答题技巧**：这是一个展示技术判断力的好题目。核心观点：**不是所有问题都需要LLM，选择合适的工具比选择最先进的工具更重要**。

### Q10: 审计日志怎么设计？为什么用WORM模式？

**简洁版**：
WORM（Write Once Read Many）模式确保审计日志一旦写入就不可修改或删除，满足HIPAA对审计追踪不可篡改的要求。

**详细版**：

**审计日志表设计**：

```sql
-- docker/init-db.sql
CREATE TABLE audit_logs (
    id BIGSERIAL PRIMARY KEY,
    session_id VARCHAR(64) NOT NULL,
    timestamp TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    agent_name VARCHAR(32) NOT NULL,
    action VARCHAR(64) NOT NULL,
    input_summary TEXT,
    output_summary TEXT,
    phi_detected BOOLEAN DEFAULT FALSE,
    phi_types TEXT[],
    compliance_result JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- HIPAA要求：审计日志不可删除
-- 通过数据库权限控制，应用只有INSERT权限，没有UPDATE/DELETE权限
-- REVOKE DELETE, UPDATE ON audit_logs FROM app_user;
```

**WORM模式的实现层次**：

| 层次 | 机制 | 说明 |
|------|------|------|
| **应用层** | 只调用INSERT | 代码中不暴露UPDATE/DELETE接口 |
| **数据库层** | 权限控制 | 应用账户只有INSERT和SELECT权限 |
| **存储层** | 合规存储 | AWS S3 Object Lock等 |
| **备份** | 6年保留 | HIPAA要求最少保留6年 |

**为什么6年？**
HIPAA要求covered entities保留审计记录至少6年。部分州法规要求更长（如加州要求10年）。

**答题技巧**：WORM是数据合规的核心概念。能说出"应用层INSERT-only + 数据库层权限控制 + 存储层Object Lock"的三层保障展示专业性。

### Q11: RBAC在医疗系统中怎么设计？

**简洁版**：
RBAC（Role-Based Access Control）基于角色控制访问权限。医疗系统中典型角色：医生、护士、编码员、管理员、审计员，每个角色有不同的数据访问权限。

**详细版**：

**角色-权限矩阵**：

| 角色 | 查看病历 | 编辑诊断 | 开具处方 | 查看编码 | 审计日志 | 系统管理 |
|------|---------|---------|---------|---------|---------|---------|
| 医生 | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| 护士 | ✅ (限本科) | ❌ | ❌ | ❌ | ❌ | ❌ |
| 编码员 | ✅ (脱敏) | ❌ | ❌ | ✅ | ❌ | ❌ |
| 审计员 | ✅ (脱敏) | ❌ | ❌ | ✅ | ✅ | ❌ |
| 管理员 | ❌ | ❌ | ❌ | ❌ | ✅ | ✅ |

**最小权限原则**：
- 编码员只能看到脱敏后的病历（不需要看真实姓名来编码）
- 审计员可以看审计日志但不能修改
- 管理员可以管理系统但不能看病历内容

**答题技巧**：用角色-权限矩阵来回答，清晰直观。强调"最小权限原则"和"职责分离"。

### Q12: 医疗数据加密有哪些层次？

### Q13: LLM在医疗领域的幻觉问题怎么应对？

**简洁版**：
幻觉是LLM生成不存在事实的现象。医疗场景中幻觉特别危险。应对策略：RAG/GraphRAG提供事实约束、结构化输出校验、规则引擎兜底、Human-in-the-loop。

**详细版**：

**医疗领域幻觉的典型表现**：

| 幻觉类型 | 示例 | 风险等级 |
|----------|------|---------|
| **虚构疾病** | 生成不存在的疾病名称 | 高 |
| **错误编码** | ICD-10编码不存在或不对应 | 高 |
| **虚假药物** | 推荐不存在的药物 | 极高 |
| **错误剂量** | 药物剂量严重偏差 | 极高 |
| **虚假交互** | 虚构药物-药物交互 | 中 |
| **不完整诊断** | 遗漏重要鉴别诊断 | 高 |

**本项目的四层防幻觉策略**：

| 层级 | 策略 | 实现 |
|------|------|------|
| **L1: 知识约束** | GraphRAG提供结构化知识 | 症状→疾病→ICD-10映射 |
| **L2: 输出校验** | Pydantic模型 + ICD-10库校验 | 编码必须在合法库中存在 |
| **L3: 规则兜底** | 确定性规则引擎 | Audit Agent不用LLM |
| **L4: 人工复核** | Human-in-the-loop | 置信度低时暂停等待医生确认 |

```python
# L2: ICD-10编码校验
def validate_icd10_code(code: str) -> bool:
    """校验LLM生成的ICD-10编码是否合法"""
    result = icd10_service.lookup_icd10(code)
    return result is not None

# 如果LLM生成了不存在的编码，标记为需要人工复核
if not validate_icd10_code(coding_result["icd10_code"]):
    state.errors.append(f"ICD-10编码{coding_result['icd10_code']}未找到，需人工确认")
```

**答题技巧**：医疗AI面试必问题。用"四层防幻觉"的结构化回答展示系统性思维。举出具体的幻觉类型（虚构疾病、错误编码）展示领域理解。

### Q14: RAG和GraphRAG在医疗领域的区别？

**简洁版**：
普通RAG基于向量相似度检索，GraphRAG基于图结构的关联检索。医疗领域GraphRAG更有优势，因为医学知识天然具有图结构（症状→疾病→药物→禁忌症）。

**详细版**：

| 维度 | 普通RAG | GraphRAG |
|------|---------|----------|
| **检索方式** | 向量相似度 | 图遍历（多跳查询） |
| **知识结构** | 扁平文档 | 实体-关系图 |
| **关联发现** | 弱（只看相似度） | 强（可以发现间接关联） |
| **推理能力** | 一跳检索 | 多跳推理（A→B→C→D） |
| **可解释性** | 弱（为什么检索到这段？） | 强（通过哪些关系链路到达？） |

**医疗领域GraphRAG的优势**：

```
场景：患者症状"持续咳嗽+发热+胸痛"

普通RAG：
  检索到"咳嗽常见于感冒、肺炎、支气管炎"
  但无法关联"胸痛"和"肺炎"的关系

GraphRAG：
  咳嗽 → 肺炎(相关度0.8)
  发热 → 肺炎(相关度0.9)
  胸痛 → 肺炎(相关度0.7)
  → 三个症状都指向肺炎，置信度很高
  → 进一步检索：肺炎 → ICD-10: J18.9
  → 进一步检索：肺炎 → 禁忌药物、推荐检查...
```

**本项目的GraphRAG实现**：

```python
# python/src/services/graphrag_service.py
SYMPTOM_DISEASE_MAP = {
    "咳嗽": [("肺炎", 0.8), ("支气管炎", 0.7), ("感冒", 0.6)],
    "发热": [("肺炎", 0.9), ("感冒", 0.8), ("流感", 0.7)],
    "胸痛": [("肺炎", 0.7), ("心绞痛", 0.6), ("肋骨骨折", 0.3)],
    # ...
}

DISEASE_ICD10_MAP = {
    "肺炎": "J18.9",
    "高血压": "I10",
    "2型糖尿病": "E11.9",
    # ...
}
```

**可选的Neo4j在线模式**：

```cypher

MATCH (s:Symptom {name: "咳嗽"})-[:INDICATES]->(d:Disease)
      -[:CODED_AS]->(c:ICD10Code)
      -[:TREATED_BY]->(m:Medication)
RETURN d.name, c.code, m.name, m.contraindications
```

**答题技巧**：画出"症状→疾病→ICD-10→药物"的图结构，然后对比RAG的"文档→向量→相似度"的检索方式，区别一目了然。

### Q15: 药物-药物交互（DDI）检测怎么实现？

**简洁版**：
维护药物交互规则库，检查处方中任意两种药物的组合是否存在已知交互，按严重程度分级（高/中/低）。

**详细版**：

**DDI检测的核心逻辑**：

```
输入：药物列表 [药物A, 药物B, 药物C]
检查：A↔B, A↔C, B↔C 所有两两组合
输出：交互列表 [{药物对, 严重程度, 说明}]
```

**本项目的实现**：

```python
# python/src/services/drug_interaction.py
INTERACTION_DATABASE = {
    ("华法林", "阿司匹林"): {
        "severity": "high",
        "description": "增加出血风险，需密切监测INR",
        "recommendation": "避免同时使用，或调整剂量并密切监测"
    },
    ("ACE抑制剂", "钾补充剂"): {
        "severity": "high",
        "description": "可能导致高钾血症",
        "recommendation": "定期监测血钾水平"
    },
    ("他汀类", "贝特类"): {
        "severity": "medium",
        "description": "增加肌病风险",
        "recommendation": "监测CK和肌肉症状"
    },
    # ...
}

def check_interactions(medications: list[str]) -> list[dict]:
    """检查药物列表中的两两交互"""
    interactions = []
    for i in range(len(medications)):
        for j in range(i+1, len(medications)):
            pair = tuple(sorted([medications[i], medications[j]]))
            if pair in INTERACTION_DATABASE:
                interactions.append({
                    "drug1": medications[i],
                    "drug2": medications[j],
                    **INTERACTION_DATABASE[pair]
                })
    return interactions
```

**严重程度分级**：

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| **High** | 禁忌或需要特别注意 | 告警，建议更换药物 |
| **Medium** | 需要监测 | 提醒，建议增加监测指标 |
| **Low** | 理论交互，临床影响小 | 记录，不阻断 |

**答题技巧**：能说出2-3个具体的药物交互案例（华法林+阿司匹林、ACE+钾）展示你对领域的了解。强调DDI检测是确定性规则而非LLM推测。

### Q16: 医学知识图谱的标准体系有哪些？

**简洁版**：
UMLS（统一医学语言系统）、SNOMED-CT（系统化医学命名法）、RxNorm（药物标准化命名）。这些是构建医学知识图谱的基础标准。

**详细版**：

| 标准 | 全称 | 覆盖范围 | 关系 |
|------|------|---------|------|
| **UMLS** | Unified Medical Language System | 元知识图谱，整合200+医学词汇表 | 包含其他所有标准 |
| **SNOMED-CT** | Systematized Nomenclature of Medicine | 临床术语（36万+概念） | 被UMLS收录 |
| **ICD-10** | International Classification of Diseases | 疾病分类（7万+编码） | 被UMLS收录 |
| **RxNorm** | - | 药物标准化命名 | 被UMLS收录 |
| **LOINC** | Logical Observation Identifiers | 检验项目编码 | 被UMLS收录 |

**UMLS的三层架构**：

```
1. Metathesaurus（元词典）
   - 集成200+源词汇表
   - 概念统一标识（CUI）
   - 如：CUI C0032285 = "肺炎"（ICD-10 J18.9、SNOMED-CT 233604007）

2. Semantic Network（语义网络）
   - 133种语义类型（如"疾病"、"药物"、"症状"）
   - 54种语义关系（如"treats"、"causes"、"prevents"）

3. SPECIALIST Lexicon（专业词典）
   - 医学术语的词形变化
   - 支持自然语言处理
```

**答题技巧**：UMLS是医学NLP的基石。能说出"UMLS整合了200+医学词汇表，通过CUI统一标识概念"就展示了足够的领域知识。

### Q17: 临床决策支持系统（CDSS）有哪些分类？

**简洁版**：
知识型CDSS（基于规则/指南）和非知识型CDSS（基于机器学习）。我们的系统是混合型——LLM负责开放推理，规则引擎负责确定性决策。

**详细版**：

| 分类 | 方法 | 优点 | 缺点 | 示例 |
|------|------|------|------|------|
| **知识型** | 规则引擎、专家系统 | 可解释、确定性 | 难以覆盖所有情况 | 药物交互检测 |
| **非知识型** | 机器学习、深度学习 | 能发现隐含模式 | 可解释性差 | 影像诊断 |
| **混合型** | LLM + 规则引擎 | 灵活且可靠 | 架构复杂 | **本项目** |

**本项目的混合策略**：

```
开放推理层（LLM）：
├── Intake Agent → 自然语言理解和结构化
├── Diagnosis Agent → 综合推理，考虑罕见病
├── Treatment Agent → 个性化治疗方案生成
└── Coding Agent → 准确ICD-10编码匹配

确定性层（规则引擎）：
├── ICD-10 Service → 编码验证
├── DDI Service → 药物交互检测
├── HIPAA Service → PHI检测和脱敏
└── Audit Agent → 合规审计（8项结构化检查）
```

**答题技巧**：CDSS是医疗信息化面试的经典话题。说出"混合型"（LLM + 规则引擎）的设计理念是核心得分点。

### Q18: 电子病历（EMR/EHR）标准有哪些？

**简洁版**：
EMR是单一机构内的电子病历，EHR是跨机构共享的电子健康记录。标准包括HL7 FHIR、HL7 CDA、OpenEHR等。

**详细版**：

| 缩写 | 全称 | 范围 | 标准 |
|------|------|------|------|
| **EMR** | Electronic Medical Record | 单机构 | 机构内部标准 |
| **EHR** | Electronic Health Record | 跨机构 | HL7 FHIR / CDA |

**主要标准对比**：

| 标准 | 组织 | 特点 | 我们的选择 |
|------|------|------|-----------|
| **HL7 FHIR** | HL7 | RESTful API，JSON/XML | ✅ |
| HL7 CDA | HL7 | 文档模型，XML | ❌ 过于复杂 |
| OpenEHR | OpenEHR基金会 | 两层建模 | ❌ 采用率较低 |
| HL7 v2 | HL7 | 消息格式，历史遗留 | ❌ 过时 |

**为什么选FHIR？**
1. **RESTful API**：与现代Web开发兼容
2. **JSON支持**：比XML更轻量，与LLM输出格式一致
3. **资源模型**：Patient、Condition等资源定义清晰
4. **生态系统**：Python fhir.resources库支持完善
5. **行业趋势**：全球主要国家都在向FHIR迁移

**答题技巧**：EMR和EHR的区别是经常混淆的概念，说清楚"EMR单机构、EHR跨机构"就展示了基础知识。然后说明为什么选FHIR而非CDA。

### Q20: 医疗AI的偏见问题如何识别和缓解？

**简洁版**：
AI偏见来源于训练数据的不均衡（如某些疾病在特定人群中数据更多）。缓解方法：多样化数据、公平性指标监控、人工复核机制。

**详细版**：

**医疗AI偏见的常见类型**：

| 偏见类型 | 表现 | 示例 |
|----------|------|------|
| **人群偏见** | 对某些群体诊断准确率低 | 女性心脏病症状非典型被低估 |
| **疾病偏见** | 常见病准确率高，罕见病低 | 罕见病诊断经常被误判为常见病 |
| **语言偏见** | 对非标准表述理解差 | 方言描述的症状被忽略 |
| **数据偏见** | 训练数据中某类病例过多 | 高血压数据多→过度诊断高血压 |

**本项目的偏见缓解策略**：

| 策略 | 实现 |
|------|------|
| **多源知识** | GraphRAG覆盖症状-疾病的多对多映射，不偏向单一疾病 |
| **鉴别诊断** | Diagnosis Agent要求输出differential_diagnoses（鉴别诊断列表） |
| **置信度标注** | 每个诊断带有confidence score，低置信度时触发Human-in-the-loop |
| **审计追踪** | Audit Agent记录每次决策，支持事后分析偏见模式 |
| **免责声明** | 明确告知用户这是辅助工具，最终决策由医生做出 |

**答题技巧**：AI偏见是近年的热门话题，在医疗领域尤其敏感。能说出具体的偏见类型（人群偏见、疾病偏见）和缓解策略，展示你的社会责任意识。
