## 一、简历项目经历的核心原则

### 1.1 STAR-L 法则

| 要素 | 含义 | 示例 |
|------|------|------|
| **S** (Situation) | 项目背景 | 医疗临床决策效率低下 |
| **T** (Task) | 你的职责 | 负责多Agent系统设计与实现 |
| **A** (Action) | 具体行动 | 使用LangGraph编排5-Agent Pipeline |
| **R** (Result) | 量化结果 | 诊断准确率94% |
| **L** (Learning) | 技术沉淀 | 沉淀了医疗AI合规设计方法论 |

### 1.2 黄金公式

```
动词 + 技术/方法 + 具体做了什么 + 量化结果
```

**好的写法**：使用LangGraph实现5-Agent Pipeline编排，通过条件路由和状态回溯机制，将诊断准确率提升至94%

**差的写法**：参与了项目开发，使用了一些AI框架

### 1.3 数字量化技巧

| 维度 | 本项目可用数字 | 说明 |
|------|----------------|------|
| **Agent数量** | 5个Agent | Intake/Diagnosis/Treatment/Coding/Audit |
| **诊断准确率** | 94% | GraphRAG辅助诊断 |
| **ICD-10编码准确率** | 80% → 94% | Coding Agent自动编码 |
| **PHI检测项** | 18项 | HIPAA Safe Harbor去标识化 |
| **知识图谱规模** | 症状-疾病-ICD-10三层映射 | GraphRAG服务 |
| **API响应时间** | <2s（单次分析） | 全链路Pipeline |
| **合规审计覆盖** | 8项结构化检查 | Audit Agent |
| **药物交互规则** | 覆盖主要DDI规则库 | 药物-药物交互检测 |

## 二、初级版（0-1年经验）

### 2.1 通用版模板

```
项目名称：多Agent医疗临床辅助决策系统
技术栈：LangGraph / FastAPI / GraphRAG / PostgreSQL / Docker
项目角色：后端开发

- 参与设计5-Agent临床决策支持系统，覆盖接诊/诊断/治疗/编码/审计全流程
- 使用LangGraph StateGraph实现Pipeline编排，支持条件路由和状态回溯
- Diagnosis Agent集成GraphRAG医学知识图谱，诊断准确率94%
- 实现HIPAA全链路合规：18项PHI检测、数据脱敏、审计日志
- 封装ICD-10编码服务和药物交互检测（DDI）服务，提供RESTful API
```

### 2.2 逐条解析

#### Bullet 1: "参与设计5-Agent临床决策支持系统，覆盖接诊/诊断/治疗/编码/审计全流程"

**为什么这样写？**

- **"5-Agent"** 是一个精确的数字，面试官一看就知道系统复杂度
- **"接诊/诊断/治疗/编码/审计"** 枚举了完整流程，展示你对业务全貌的理解
- **"参与设计"** 对于初级开发者是合适的动词，既不夸大也不贬低

**亮点**：展示了你理解多Agent架构不是拍脑袋决定的，而是根据临床流程拆分的——面试官一定会追问"为什么分5个Agent"，你就可以展开讲**单一职责原则**和**领域驱动设计**。

**面试追问准备**：
- Q: 为什么是5个Agent而不是1个？
- A: 基于单一职责原则和临床工作流划分。每个Agent对应一个临床角色（接诊护士→主治医师→药师→编码员→审计员），职责隔离便于独立迭代和测试。

#### Bullet 2: "使用LangGraph StateGraph实现Pipeline编排，支持条件路由和状态回溯"

**为什么这样写？**

- **"LangGraph StateGraph"** 精确到框架的具体组件，展示你不是只会用而是理解内部机制
- **"条件路由"** 指的是 `_route_after_diagnosis` 函数，根据 `needs_more_info` 决定是回到Intake还是进入Treatment
- **"状态回溯"** 指的是 LangGraph 的 Checkpointer 机制，支持从任意节点恢复

**亮点**：LangGraph是2024-2025年最火的Agent编排框架，写在简历上能立刻被AI方向的面试官注意到。

#### Bullet 3: "Diagnosis Agent集成GraphRAG医学知识图谱，诊断准确率94%"

**为什么这样写？**

- **"GraphRAG"** 是区别于普通RAG的高级技术，展示你理解图结构化检索的优势
- **"94%"** 是一个具体的量化数字，比"提高了准确率"强10倍
- **"医学知识图谱"** 点明了领域特殊性，暗示你处理过非通用场景

**亮点**：这一条同时展示了技术深度（GraphRAG）和业务理解（医学知识图谱）。

#### Bullet 4: "实现HIPAA全链路合规：18项PHI检测、数据脱敏、审计日志"

**为什么这样写？**

- **"HIPAA"** 是医疗行业核心法规，展示你有领域合规意识
- **"18项PHI"** 对应 Safe Harbor 方法的 18 种受保护健康信息标识符
- **"全链路"** 说明合规不是单点检查，而是贯穿整个Pipeline

**亮点**：安全与合规是高级话题，初级开发者能写出这一条说明综合素质高。

#### Bullet 5: "封装ICD-10编码服务和药物交互检测（DDI）服务，提供RESTful API"

**为什么这样写？**

- **"ICD-10"** 和 **"DDI"** 是医疗信息化的核心概念，展示领域知识
- **"封装...服务"** 说明你能独立设计模块
- **"RESTful API"** 是最基础但必须有的技术关键词

## 三、中级版（1-3年经验）

### 3.1 模板（强调架构设计）

```
项目名称：多Agent医疗临床辅助决策系统（三语言实现）
技术栈：Python(LangGraph/FastAPI) + Java(Spring Boot/LangGraph4j) + Go(Gin)
项目角色：系统架构设计 & 核心开发

- 主导设计多Agent临床决策架构，基于领域驱动将临床流程拆分为5个自治Agent，
  实现Intake→Diagnosis→Treatment→Coding→Audit全链路自动化
- 设计ClinicalState共享状态机制，使用Annotated类型实现消息自动合并，
  Agent间通过不可变状态传递实现松耦合通信
- 实现Diagnosis→Intake条件回退路由，设置最大重试2次的降级策略，
  避免LLM幻觉导致的无限循环
- 构建GraphRAG医学知识图谱服务，支持Neo4j在线/内存离线双模式，
  症状→疾病→ICD-10三层关联检索，诊断召回率显著提升
- 设计确定性Audit Agent（非LLM），正则驱动18项PHI扫描+数据脱敏，
  确保合规审计的可靠性和可解释性
- 主导Python/Java/Go三语言实现并保持架构一致性，沉淀跨语言
  Agent设计模式（Python StateGraph / Java手动编排 / Go接口抽象）
```

### 3.2 逐条解析

#### Bullet 1: "主导设计多Agent临床决策架构..."

**中级 vs 初级的区别**：
- 动词从"参与设计"升级为"**主导设计**"
- 增加了"**基于领域驱动**"的方法论说明
- **"自治Agent"** 展示你理解Agent的自主性特征
- **"全链路自动化"** 强调端到端能力

**面试官看到这条会想**："这个人不只是写代码，还能做架构决策。"

#### Bullet 2: "设计ClinicalState共享状态机制..."

**为什么这样写？**

- **"ClinicalState"** 直接点出核心数据结构名，展示你对代码非常熟悉
- **"Annotated类型实现消息自动合并"** 指的是 LangGraph 中 `Annotated[list[BaseMessage], add_messages]` 的机制
- **"不可变状态传递"** 展示你理解函数式编程思想和Agent间通信设计

**技术深度加分**：这一条涉及了类型系统、状态管理、通信模式三个维度。

#### Bullet 3: "实现Diagnosis→Intake条件回退路由..."

**为什么强调"最大重试2次"？**

- 这是一个**工程化的细节**，展示你考虑了异常情况
- LLM可能因为幻觉持续返回 `needs_more_info=true`，不设上限会导致死循环
- Java和Go版本中有显式的 `MAX_DIAGNOSIS_RETRIES = 2`
- 这正好是面试官爱问的"如何处理LLM不稳定性"的完美答案

#### Bullet 4: "构建GraphRAG医学知识图谱服务..."

**中级版的进阶之处**：
- 提到了**"双模式"**（Neo4j在线 + 内存离线），展示你考虑了不同部署环境
- **"三层关联检索"** 比单纯说"知识图谱"更具体
- 展示了你对RAG技术栈的深入理解

#### Bullet 5: "设计确定性Audit Agent（非LLM）..."

**为什么要特意强调"非LLM"？**

- 这展示了一个关键的架构决策：**不是所有Agent都需要LLM**
- 合规审计需要**确定性**和**可解释性**，用规则引擎比LLM更合适
- 面试官看到这条会认为你有独立的技术判断力

#### Bullet 6: "主导Python/Java/Go三语言实现..."

**中级版独有的亮点**：
- **"三语言实现"** 直接说明你是全栈能力
- **"架构一致性"** 展示你能抽象出语言无关的设计模式
- 具体列出了三种语言的不同实现方式，展示你理解各语言的特点
