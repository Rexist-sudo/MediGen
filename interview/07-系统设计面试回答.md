## 第一阶段：需求分析（5分钟）

### 面试脚本

> 在开始设计之前，我想先确认一下需求范围和约束条件。

### 1.1 功能需求

> 首先是功能需求。我理解的"临床辅助决策系统"需要覆盖临床工作流的核心环节：

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **患者信息采集** | 从自然语言描述中提取和结构化患者信息 | P0 |
| **辅助诊断** | 基于症状和病史给出诊断建议和鉴别诊断 | P0 |
| **治疗方案生成** | 根据诊断生成治疗方案，包括药物和非药物治疗 | P0 |
| **ICD-10编码** | 自动为诊断结果匹配ICD-10编码和DRG分组 | P0 |
| **合规审计** | HIPAA合规检查，PHI检测和脱敏 | P0 |
| **药物交互检测** | 检查处方中的药物-药物交互风险 | P1 |
| **FHIR数据转换** | 支持与外部EHR系统的FHIR R4标准互操作 | P1 |
| **影像分析** | 医学影像辅助分析 | P2（未来） |

> 请问面试官，这些功能的理解是否正确？有没有特别需要关注的功能？

### 1.2 非功能需求

> 接下来是非功能需求：

| 需求 | 指标 | 说明 |
|------|------|------|
| **延迟** | 单次分析<5s，目标<2s | 5个Agent串行执行的总延迟 |
| **准确率** | 诊断≥90%，编码≥90% | 作为辅助工具的可接受水平 |
| **可用性** | 99.9%（三个9） | 允许每年约8小时不可用 |
| **合规性** | HIPAA全链路合规 | 医疗数据必须满足 |
| **可扩展性** | 支持新增Agent | 未来可能增加影像分析等Agent |
| **并发** | 100+并发用户 | 中等规模医院 |
| **数据量** | 日均1000+次分析 | 中等负载 |

### 1.3 边界和约束

> 几个关键约束：

| 约束 | 说明 |
|------|------|
| 定位是"辅助"而非"替代" | 最终决策权在医生 |
| 依赖外部LLM API | OpenAI/Azure，不自建模型 |
| 需要支持中文 | 患者描述和诊断结果都是中文 |
| 不涉及影像和基因 | 第一版只处理文本数据 |

### 1.4 需求确认话术

> "在继续之前，我想确认几个关键假设：
> 1. 我们的系统是辅助工具，最终决策由医生做出，对吗？
> 2. 我们使用云端LLM API（如OpenAI），不需要自己训练模型，对吗？
> 3. 第一版专注文本处理，影像和基因数据是未来扩展，对吗？
> 4. HIPAA合规是硬性要求，对吗？
>
> 好的，确认了这些之后，让我开始设计。"

### 2.2 架构说明

> 让我解释一下每层的职责：

**客户端层**：
> 可以是医院的Web前端、移动端APP，或者通过API与现有EHR系统集成。我们只设计后端服务，前端不在范围内。

**API网关层**：
> 所有请求先经过网关。负责JWT认证验证身份、RBAC授权控制权限（医生可以发起分析，编码员只能查询编码）、限流防止单用户消耗过多LLM配额、全局日志记录。

**Agent编排层（核心）**：
> 这是系统的心脏。5个Agent通过共享状态（ClinicalState）协作，由编排引擎控制执行顺序和条件路由。每个Agent有明确的职责，对应临床流程中的一个角色。关键设计是Diagnosis→Intake的条件回退——如果诊断Agent认为信息不足，会回到采集Agent补充信息。

**服务与数据层**：
> 提供Agent需要的基础能力。GraphRAG服务连接Neo4j提供医学知识图谱查询。ICD-10服务提供编码检索和验证。DDI服务提供药物交互检测。FHIR服务负责数据格式转换。PostgreSQL存储临床会话和审计日志。Redis做缓存。LLM API是外部依赖。

### 2.3 技术选型

| 组件 | 技术选择 | 选型理由 |
|------|---------|---------|
| 编排引擎 | LangGraph StateGraph | 支持循环和条件路由，状态管理完善 |
| Web框架 | FastAPI | 异步支持好，自动OpenAPI文档 |
| 数据库 | PostgreSQL | JSONB支持半结构化数据，审计合规 |
| 图数据库 | Neo4j | 医学知识图谱的天然选择 |
| 缓存 | Redis | 高性能KV，会话管理 |
| LLM | OpenAI gpt-4o-mini | 性价比高，结构化输出好 |
| 容器化 | Docker Compose | 开发环境一键启动 |

### 3.3 错误响应设计

```json
{
    "error": "AGENT_TIMEOUT",
    "detail": "Diagnosis Agent执行超时（>10s）",
    "agent": "diagnosis",
    "session_id": "session_001",
    "partial_result": {
        "patient_info": {...},
        "diagnosis": null,
        "treatment_plan": null,
        "coding": null,
        "audit": null
    }
}
```

> 一个关键的设计决策：即使某个Agent失败，我们也返回已完成部分的结果（partial_result），而不是整体返回500。在医疗场景中，部分结果（附带警告）比完全没有结果要好。

## 第四阶段：数据模型设计（5分钟）

### 面试脚本

> 接下来是数据模型设计，分为两部分：Pipeline内部的状态模型和持久化的数据库模型。

### 4.1 ClinicalState（Pipeline内部状态）

```python
class ClinicalState(BaseModel):
    """Pipeline全流程的共享状态"""
    
    # 输入
    raw_input: str = ""
    
    # Intake Agent输出
    patient_info: dict | None = None
    
    # Diagnosis Agent输出
    diagnosis: dict | None = None
    needs_more_info: bool = False  # 条件路由关键字段
    
    # Treatment Agent输出
    treatment_plan: dict | None = None
    
    # Coding Agent输出
    coding_result: dict | None = None
    
    # Audit Agent输出
    audit_result: dict | None = None
    
    # 元数据
    messages: Annotated[list[BaseMessage], add_messages] = []
    errors: list[str] = []
    current_agent: str = ""
    degraded_agents: list[str] = []
```

> **设计要点**：
> 1. 字段按Agent职责划分，每个Agent只写自己的字段
> 2. `needs_more_info`是条件路由的驱动字段
> 3. `messages`使用Annotated实现自动追加
> 4. `errors`累积记录，不中断Pipeline
> 5. 大部分字段初始为None，随Pipeline推进逐步填充

### 4.2 数据库模型

```sql
-- 临床会话表
CREATE TABLE clinical_sessions (
    id VARCHAR(64) PRIMARY KEY,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    status VARCHAR(16) NOT NULL DEFAULT 'pending',
    -- JSONB存储完整的ClinicalState快照
    state JSONB NOT NULL,
    -- 冗余关键字段用于查询
    primary_diagnosis_code VARCHAR(16),
    patient_age INTEGER,
    pipeline_duration_ms INTEGER
);

-- 审计日志表（WORM模式）
CREATE TABLE audit_logs (
    id BIGSERIAL PRIMARY KEY,
    session_id VARCHAR(64) NOT NULL REFERENCES clinical_sessions(id),
    timestamp TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    agent_name VARCHAR(32) NOT NULL,
    action VARCHAR(64) NOT NULL,
    input_summary TEXT,       -- 脱敏后的输入摘要
    output_summary TEXT,      -- 脱敏后的输出摘要
    phi_detected BOOLEAN DEFAULT FALSE,
    phi_types TEXT[],          -- 检测到的PHI类型数组
    compliance_result JSONB,  -- 结构化合规检查结果
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
    -- 注意：没有updated_at，因为WORM模式不允许更新
);

-- 索引
CREATE INDEX idx_sessions_created ON clinical_sessions(created_at);
CREATE INDEX idx_sessions_diagnosis ON clinical_sessions(primary_diagnosis_code);
CREATE INDEX idx_audit_session ON audit_logs(session_id);
CREATE INDEX idx_audit_timestamp ON audit_logs(timestamp);
CREATE INDEX idx_audit_compliance ON audit_logs USING GIN(compliance_result);
```

> **选择JSONB的理由**：ClinicalState的结构可能随Agent迭代变化，JSONB允许灵活查询而不需要频繁ALTER TABLE。同时PostgreSQL的GIN索引支持JSONB的高效查询。

### 4.3 数据流向图

```
用户输入
    ↓
ClinicalState (内存，Pipeline运行时)
    ↓
clinical_sessions (PostgreSQL，持久化)
    ↓
audit_logs (PostgreSQL，WORM审计)
    ↓
FHIR Resources (JSON，可选导出到外部EHR)
```

### 5.1 Pipeline流程

```
[START]
    │
    ▼
┌──────────────┐
│ Intake Agent │ ← 解析患者描述，提取结构化信息
│ (T=0.1)      │

│  (T=0.2)     │     └──────────────┘

       │                    NO
       ▼                    │
┌──────────────┐←───────────┘
│  Treatment   │ ← 生成治疗方案+药物交互检测
│  Agent       │
│  (T=0.2)     │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   Coding     │ ← ICD-10编码+DRG分组
│   Agent      │
│   (T=0.1)    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   Audit      │ ← 合规审计（规则引擎，非LLM）
│   Agent      │
│   (无LLM)    │
└──────┬───────┘
       │
       ▼
```

### 5.2 每个Agent的详细设计

#### Intake Agent

| 维度 | 说明 |
|------|------|
| **职责** | 从自然语言中提取结构化患者信息 |
| **输入** | `state.raw_input`（患者描述文本） |
| **输出** | `state.patient_info`（结构化PatientInfo） |
| **LLM** | gpt-4o-mini, temperature=0.1 |
| **Prompt策略** | 要求输出固定JSON Schema，包含name/age/gender/symptoms/history |
| **错误处理** | JSON解析失败时重试一次，仍失败则记录错误 |

```python
INTAKE_SYSTEM_PROMPT = """你是一位经验丰富的接诊护士。
请从患者描述中提取以下信息，以JSON格式返回：
{
    "patient_name": "姓名",
    "age": 数字,
    "gender": "male/female",
    "chief_complaint": "主诉",
    "symptoms": ["症状1", "症状2"],
    "medical_history": ["病史1"],
    "current_medications": ["用药1"],
    "allergies": ["过敏1"]
}
只返回JSON，不要其他文字。"""
```

#### Diagnosis Agent

| 维度 | 说明 |
|------|------|
| **职责** | 基于患者信息做出诊断建议 |
| **输入** | `state.patient_info` |
| **输出** | `state.diagnosis` + `state.needs_more_info` |
| **LLM** | gpt-4o-mini, temperature=0.2 |
| **辅助** | GraphRAG服务提供结构化知识约束 |
| **关键字段** | `needs_more_info: bool` 驱动条件路由 |

> Diagnosis Agent是最关键的Agent，它有两个特殊设计：
>
> 第一，**GraphRAG辅助**。在调用LLM之前，先用GraphRAG服务查询症状→疾病的映射关系，把结构化的知识作为额外上下文传给LLM，减少幻觉风险。
>
> 第二，**条件回退**。如果诊断Agent认为当前信息不足以做出可靠诊断，它设置`needs_more_info=true`，Pipeline会回退到Intake Agent让用户补充信息。但这里有一个关键设计——**最大重试2次**，防止LLM幻觉导致的死循环。

```python
def _route_after_diagnosis(state: ClinicalState) -> str:
    if state.needs_more_info:
        return "intake"     # 回退补充信息
    return "treatment"      # 继续治疗

# Java/Go版本通过MAX_DIAGNOSIS_RETRIES=2做硬限制
```

#### Treatment Agent

| 维度 | 说明 |
|------|------|
| **职责** | 生成治疗方案（药物+非药物） |
| **输入** | `state.patient_info` + `state.diagnosis` |
| **输出** | `state.treatment_plan` |
| **LLM** | gpt-4o-mini, temperature=0.2 |
| **辅助** | DDI服务检测药物交互 |

> Treatment Agent在生成药物方案后，应该调用DDI服务做交叉检查。如果发现高风险交互（如华法林+阿司匹林），要在治疗方案中标注警告或建议替代药物。

#### Coding Agent

| 维度 | 说明 |
|------|------|
| **职责** | 为诊断匹配ICD-10编码和DRG分组 |
| **输入** | `state.diagnosis` + `state.treatment_plan` |
| **输出** | `state.coding_result` |
| **LLM** | gpt-4o-mini, temperature=0.1 |
| **辅助** | ICD-10服务做编码验证 |

> 低temperature（0.1）是因为编码需要精确匹配。LLM生成的ICD-10编码必须与内置编码库交叉验证——如果编码不存在，标记为需要人工复核。

#### Audit Agent

| 维度 | 说明 |
|------|------|
| **职责** | HIPAA合规审计、PHI检测和脱敏 |
| **输入** | Pipeline全量输出 |
| **输出** | `state.audit_result` |
| **LLM** | **不使用LLM** |
| **引擎** | 正则规则引擎 |

> **这是一个重要的架构决策**——Audit Agent不使用LLM，而是用确定性的规则引擎。原因有三：
> 1. 合规审计需要100%确定性，LLM存在幻觉风险
> 2. 合规审计需要100%可解释性，能说出"为什么标记这段文本为PHI"
> 3. 审计结果用于法律合规，必须可审计和可复现

### 5.3 LangGraph编排代码

```python
from langgraph.graph import StateGraph, START, END

workflow = StateGraph(ClinicalState)

# 添加5个节点
workflow.add_node("intake", intake_agent)
workflow.add_node("diagnosis", diagnosis_agent)
workflow.add_node("treatment", treatment_agent)
workflow.add_node("coding", coding_agent)
workflow.add_node("audit", audit_agent)

# 固定边
workflow.add_edge(START, "intake")
workflow.add_edge("intake", "diagnosis")
workflow.add_edge("treatment", "coding")
workflow.add_edge("coding", "audit")
workflow.add_edge("audit", END)

# 条件边：Diagnosis后根据状态路由
workflow.add_conditional_edges(
    "diagnosis",
    _route_after_diagnosis,
    {"intake": "intake", "treatment": "treatment"},
)

# 编译
app = workflow.compile(checkpointer=MemorySaver())
```

## 第六阶段：知识图谱设计（4分钟）

### 面试脚本

> 接下来说说知识图谱的设计。这是提升诊断准确率的关键组件。

### 6.1 图模型设计

```
┌──────────┐    INDICATES    ┌──────────┐    CODED_AS    ┌──────────┐
│  Symptom │───────────────→│  Disease │───────────────→│ ICD10Code│
│  (症状)   │  (关联度0-1)   │  (疾病)   │               │  (编码)   │
└──────────┘                └──────┬───┘                └──────────┘
                                   │
                            TREATED_BY
                                   │
                                   ▼
                            ┌──────────┐   INTERACTS_WITH   ┌──────────┐
                            │   Drug   │←──────────────────→│   Drug   │
                            │  (药物)   │    (药物交互)       │  (药物)   │
                            └──────────┘                    └──────────┘
```

**节点类型**：

| 节点 | 属性 | 示例 |
|------|------|------|
| Symptom | name, severity_levels | {name: "咳嗽", severity_levels: ["轻", "中", "重"]} |
| Disease | name, category, prevalence | {name: "肺炎", category: "呼吸系统", prevalence: 0.03} |
| ICD10Code | code, description, chapter | {code: "J18.9", description: "肺炎，未特指"} |
| Drug | name, class, dosage_range | {name: "阿莫西林", class: "抗生素"} |

**关系类型**：

| 关系 | 属性 | 说明 |
|------|------|------|
| INDICATES | confidence (0-1) | 症状指向疾病的关联度 |
| CODED_AS | primary (bool) | 是否为主编码 |
| TREATED_BY | line (1st/2nd/3rd) | 一线/二线/三线治疗 |
| INTERACTS_WITH | severity (high/med/low) | 交互严重程度 |

### 6.2 查询示例

```cypher
-- 从症状查找可能的疾病和ICD-10编码
MATCH (s:Symptom)-[r:INDICATES]->(d:Disease)-[:CODED_AS]->(c:ICD10Code)
WHERE s.name IN ['咳嗽', '发热', '胸痛']
WITH d, c, SUM(r.confidence) AS total_confidence
ORDER BY total_confidence DESC
LIMIT 5
RETURN d.name, c.code, c.description, total_confidence

-- 检查药物交互
MATCH (d1:Drug)-[r:INTERACTS_WITH]->(d2:Drug)
WHERE d1.name IN ['华法林', '阿司匹林', '阿莫西林']
  AND d2.name IN ['华法林', '阿司匹林', '阿莫西林']
RETURN d1.name, d2.name, r.severity, r.description
```

### 6.3 双模式设计

> 一个重要的工程化决策：GraphRAG服务支持**在线模式**和**离线模式**。

| 模式 | 存储 | 性能 | 适用场景 |
|------|------|------|---------|
| **在线模式** | Neo4j | 查询灵活，支持复杂图遍历 | 生产环境 |
| **离线模式** | Python内存Dict | 极快，无IO | 开发/测试/Neo4j不可用时降级 |

> 离线模式用Python字典模拟图结构：

```python
SYMPTOM_DISEASE_MAP = {
    "咳嗽": [("肺炎", 0.8), ("支气管炎", 0.7), ("感冒", 0.6)],
    "发热": [("肺炎", 0.9), ("感冒", 0.8), ("流感", 0.7)],
}
DISEASE_ICD10_MAP = {
    "肺炎": "J18.9",
    "高血压": "I10",
}
```

> 这种设计让开发环境不依赖Neo4j也能正常工作，极大降低了开发门槛。

### 7.2 PHI检测与脱敏

> Audit Agent的核心功能。使用正则表达式扫描全部18项PHI：

| PHI类型 | 正则模式 | 脱敏方式 |
|---------|---------|---------|
| 手机号 | `\b1[3-9]\d{9}\b` | `[PHONE]` |
| 身份证 | `\b\d{17}[\dXx]\b` | `[ID_CARD]` |
| 邮箱 | `\b[\w.+-]+@[\w-]+\.[\w.-]+\b` | `[EMAIL]` |
| 日期 | `\b\d{4}[-/]\d{1,2}[-/]\d{1,2}\b` | `[DATE]` |
| 地址 | 地名词典 + 地址模式 | `[ADDRESS]` |
| 姓名 | 姓名词典 + 上下文 | `[NAME]` |

> **为什么用正则而不用LLM？** 合规检测需要确定性——同一个输入必须得到同一个输出。正则规则可以100%复现和审计，LLM做不到。

## 面试官常见追问及应对

### 追问1: "如果OpenAI挂了怎么办？"

> 三层降级策略：
> 1. **重试**：指数退避重试3次
> 2. **备用模型**：切换到Azure OpenAI或其他LLM提供商
> 3. **本地降级**：GraphRAG离线模式 + ICD-10内置库 + 规则引擎，可以提供基础的编码和审计服务（不依赖LLM）

### 追问2: "如何保证诊断结果不出错？"

> 四层保障：
> 1. GraphRAG提供结构化知识约束，限制LLM的推理范围
> 2. 输出结果与ICD-10编码库交叉验证
> 3. 要求输出鉴别诊断列表（不只给一个答案）
> 4. 低置信度时Human-in-the-loop，暂停等待医生确认
> 5. 最终声明：**辅助工具，不替代医生决策**

### 追问3: "100个并发用户怎么支撑？"

> 计算：
> - 单次分析约2s，5次LLM调用
> - 100并发 = 500次/s的LLM调用
> - OpenAI API限制：tier-3约5000 RPM
> - 所以100并发在API限制内
> - API服务水平扩展到5个Pod，每Pod处理20并发
> - PostgreSQL连接池100连接
> - Redis缓存热点查询

### 追问4: "数据量增长到百万级会有什么问题？"

> 数据库层面：
> - clinical_sessions表做时间分区（按月）
> - audit_logs表归档到冷存储（6年后可以移到S3）
> - 热数据用Redis缓存
> - 必要时引入读写分离（主写从读）
>
> 图数据库层面：
> - Neo4j集群模式支持水平扩展
> - 图数据量不会很大（医学知识是有限集）

### 追问5: "如何处理不同医院的定制需求？"

> 多租户架构：
> - 数据隔离：每个医院独立Schema或独立数据库
> - Prompt定制：不同医院可以配置不同的Agent Prompt（如不同的诊断偏好）
> - 编码版本：支持ICD-10不同版本（中国版CHS vs 国际版）
> - 合规规则：不同国家的合规规则不同（HIPAA vs GDPR vs 中国个保法）

### 追问6: "这个系统和市面上的CDSS有什么区别？"

> 三个核心差异：
> 1. **多Agent架构**：传统CDSS是单一规则引擎或单一ML模型，我们是多Agent协作
> 2. **LLM + 规则引擎混合**：传统CDSS要么纯规则要么纯ML，我们是LLM做推理+规则引擎做兜底
> 3. **端到端覆盖**：传统CDSS通常只做诊断或只做编码，我们覆盖接诊→诊断→治疗→编码→审计全流程

### 追问7: "如果给你更多时间，你还会做什么？"

> 按优先级：
> 1. **集成测试框架**：用Mock LLM做端到端自动测试
> 2. **LangSmith集成**：Agent级别的追踪和评估
> 3. **A/B测试框架**：对比不同Prompt版本的效果
> 4. **影像分析Agent**：扩展到医学影像
> 5. **多语言支持**：国际化（英语、日语等）
> 6. **联邦学习**：多医院数据协作训练而不共享原始数据
