## 1. 30秒项目介绍

MediGenius 是一个多 Agent 医疗临床辅助决策系统。它把一次临床辅助决策拆成五个专业角色：患者信息采集、鉴别诊断、治疗推荐、医学编码和合规审计。系统接收患者自然语言描述，经过 LangGraph 编排的 Agent Pipeline，输出结构化患者信息、诊断证据链、治疗建议、ICD-10/DRGs 编码结果以及 HIPAA 合规审计报告。

项目的技术重点不是简单调用大模型，而是把 LLM、GraphRAG 知识图谱、FHIR 医疗互操作标准、ICD-10 编码和 HIPAA 合规规则组合成一个可审计、可测试、可扩展的工程系统。

---

## 2. 1分钟面试版讲法

这个项目解决的是医疗场景中“从患者描述到临床辅助决策报告”的端到端流程。传统单一 Prompt 很难同时完成信息抽取、诊断、治疗、编码和合规检查，而且可测试性和可审计性都比较差。所以我采用多 Agent Pipeline 架构，将任务拆成 Intake、Diagnosis、Treatment、Coding、Audit 五个 Agent。

核心架构分为四层：

| 层级 | 说明 |
|------|------|
| API 层 | FastAPI / Spring Boot / Gin 提供 REST 接口 |
| Agent Pipeline 层 | LangGraph 或显式 Pipeline 编排 5 个 Agent |
| 服务层 | GraphRAG、FHIR、HIPAA、ICD-10、DDI 等可复用服务 |
| 基础设施层 | PostgreSQL、Neo4j、Redis、OpenAI API |

项目中最关键的设计是：医疗场景需要确定性和可审计性，所以我选择 Pipeline 而不是 Debate 或 Supervisor；诊断环节接入 GraphRAG 降低幻觉；合规环节不使用 LLM，而是用正则规则和 WORM 审计日志保证可复现；对外输出则映射为 FHIR R4 资源，便于和 HIS/EMR 系统集成。

## 4. 项目背景和业务价值

医疗临床辅助决策不是单一任务，而是一个多阶段流程：

1. 从患者口述中提取症状、病史、用药、过敏、生命体征等信息。
2. 根据症状和检查结果做鉴别诊断。
3. 给出治疗建议，并检查药物相互作用和过敏禁忌。
4. 将诊断映射为 ICD-10 编码，并做 DRGs 分组。
5. 检查输出是否包含 PHI，生成 HIPAA 合规审计结果。

这个场景适合多 Agent，因为每个环节对应不同专业角色，输入输出边界清晰，且医疗系统需要审计、追溯和合规。

面试可以这样说：

> 我不是把所有任务塞进一个超长 Prompt，而是把临床决策拆成多个专家 Agent。这样每个 Agent 的 Prompt 更短、职责更清晰、测试更独立，出错时也能定位到具体环节。

## 6. 整体架构

系统采用四层架构：

```text
客户端 / Web UI / curl
        |
        v
API 层：FastAPI / Spring Boot / Gin
        |
        v
Agent Pipeline 层：Intake -> Diagnosis -> Treatment -> Coding -> Audit
        |
        +--> LLM：OpenAI GPT-4o-mini
        +--> 服务层：GraphRAG / DDI / ICD-10 / FHIR / HIPAA
        |
        v
基础设施层：PostgreSQL / Neo4j / Redis
```

架构讲解重点：

- API 层只负责请求响应，不直接写复杂业务。
- Agent Pipeline 层负责编排流程和状态流转。
- 服务层封装确定性的能力，比如 ICD-10 查询、FHIR 转换、PHI 检测、DDI 检查。
- 基础设施层提供持久化、缓存、图谱和 LLM 能力。

面试时可以强调：

> 这个分层的核心目的是降低耦合。Agent 不直接操作数据库，也不把所有逻辑写进 Prompt，确定性能力尽量下沉到服务层，LLM 只负责更适合它的自然语言理解和推理。

## 7. 为什么选择 Pipeline 编排

项目对比了 Pipeline、Supervisor、Debate、Voting 四种模式，最终选择 Pipeline。

| 模式 | 优点 | 缺点 | 是否适合本项目 |
|------|------|------|----------------|
| Pipeline | 流程确定、成本固定、可审计 | 灵活性有限 | 适合 |
| Supervisor | 动态调度灵活 | 中心调度器复杂且不可控 | 不优先 |
| Debate | 多视角减少偏差 | 成本高、延迟高、审计难 | 不适合临床主流程 |
| Voting | 关键决策可靠性更高 | LLM 调用成本高 | 可作为未来增强 |

选择 Pipeline 的原因：

1. 医疗流程天然是“采集 -> 诊断 -> 治疗 -> 编码 -> 审计”。
2. 合规场景要求每一步可追溯。
3. LLM 调用次数固定，成本可控。
4. 方便定位某个 Agent 的错误。

项目不是完全线性的死流程。Diagnosis Agent 可以设置 `needs_more_info = true`，Pipeline 通过条件路由回到 Intake Agent 补充信息，并限制最大重试次数，防止无限循环。

```text
Intake -> Diagnosis -> Treatment -> Coding -> Audit
             |
             +-- needs_more_info=true 且 retries<2 -> Intake
```

## 8. LangGraph 核心设计

Python 版本使用 LangGraph 的核心概念：

| 概念 | 项目对应 | 说明 |
|------|----------|------|
| State | `ClinicalState` | 所有 Agent 共享的数据对象 |
| Node | 5 个 Agent 函数 | 每个节点接收 State，返回要更新的字段 |
| Edge | 固定执行顺序 | 定义 Intake 到 Diagnosis 等普通边 |
| Conditional Edge | Diagnosis 后的条件路由 | 根据 `needs_more_info` 决定回环或继续 |
| Checkpointer | `MemorySaver` | 支持会话状态保存和恢复 |

`ClinicalState` 是系统核心状态模型：

```text
raw_input           外部输入，患者自然语言描述
patient_info        Intake 写入，Diagnosis/Treatment/Audit 读取
diagnosis           Diagnosis 写入，Treatment/Coding/Audit 读取
needs_more_info     Diagnosis 写入，Pipeline 路由读取
treatment_plan      Treatment 写入，Coding/Audit 读取
coding_result       Coding 写入，Audit 读取
audit_result        Audit 写入，对外输出
errors              任意 Agent 写入，记录错误
current_agent       标记当前执行节点
```

设计原则：

- Agent 之间不直接通信，只通过 State 传递数据。
- 每个核心字段只有一个 Agent 写入。
- 后续 Agent 只读前序结果，不修改前序输出。
- 使用 Pydantic 保证结构和类型。

## 9. 五大 Agent 设计

### 9.1 Intake Agent

| 项目 | 说明 |
|------|------|
| 角色 | 问诊护士 |
| 输入 | `raw_input` |
| 输出 | `patient_info` |
| LLM | 使用，temperature=0.1 |
| 重点 | 从非结构化患者描述中提取结构化信息 |

核心输出字段包括：

- 姓名、年龄、性别。
- 主诉、症状列表、持续时间、严重程度。
- 既往病史、家族史、过敏史、当前用药。
- 生命体征和实验室检查结果。

技术重点：

- Prompt 中给出完整 JSON Schema。
- 明确要求只输出 JSON，减少解析失败。
- 字段设计参考 FHIR Patient、Observation、AllergyIntolerance 等资源。

### 9.2 Diagnosis Agent

| 项目 | 说明 |
|------|------|
| 角色 | 诊断医生 |
| 输入 | `patient_info` |
| 输出 | `diagnosis`、`needs_more_info` |
| LLM | 使用，temperature=0.2 |
| 重点 | 鉴别诊断、置信度、证据链、信息不足检测 |

核心能力：

- 根据症状、病史、检查结果生成主诊断和鉴别诊断。
- 给每个候选诊断提供置信度。
- 接入 GraphRAG，先从知识图谱召回候选疾病，再让 LLM 结合临床信息推理。
- 如果信息不足，则设置 `needs_more_info`，触发回环。

面试可以强调：

> Diagnosis Agent 不是完全相信 LLM。它会先利用 GraphRAG 根据症状召回候选疾病，再把候选列表注入 Prompt，降低幻觉和遗漏。

### 9.3 Treatment Agent

| 项目 | 说明 |
|------|------|
| 角色 | 临床药剂师 |
| 输入 | `patient_info` + `diagnosis` |
| 输出 | `treatment_plan` |
| LLM | 使用，temperature=0.2 |
| 重点 | 循证治疗、药物交互、过敏禁忌 |

核心能力：

- 生成药物治疗和非药物治疗方案。
- 检查 Drug-Drug Interaction。
- 检查过敏史和禁忌症。
- 给出随访计划、风险提示和证据来源。

DDI 示例：

| 药物组合 | 风险 |
|----------|------|
| warfarin + aspirin | 出血风险增加 |
| metformin + contrast dye | 乳酸酸中毒风险 |
| SSRI + MAOI | 5-HT 综合征风险 |
| ACE inhibitor + potassium supplement | 高钾血症 |

### 9.4 Coding Agent

| 项目 | 说明 |
|------|------|
| 角色 | 医学编码员 |
| 输入 | `diagnosis` + `treatment_plan` |
| 输出 | `coding_result` |
| LLM | 使用，temperature=0.1 |
| 重点 | ICD-10-CM 编码和 DRGs 分组 |

核心能力：

- 将诊断映射为 ICD-10-CM 编码。
- 输出主诊断编码和次要诊断编码。
- 根据 ICD-10 前缀做 DRGs 分组，输出权重和平均住院日。
- 给出编码置信度，低置信度时提示人工复核。

面试时可以说：

> 编码是高确定性任务，所以 temperature 设置更低，并且可以通过内置 ICD-10 库对 LLM 输出做校验，避免模型编造不存在的编码。

### 9.5 Audit Agent

| 项目 | 说明 |
|------|------|
| 角色 | 合规审计官 |
| 输入 | 全部 State 字段 |
| 输出 | `audit_result` |
| LLM | 不使用 |
| 重点 | PHI 检测、脱敏、HIPAA 检查、审计日志 |

这是项目中非常重要的设计取舍：Audit Agent 故意不用 LLM。

原因：

- 合规检查必须可重复、可解释、可审计。
- LLM 可能遗漏 PHI，也可能幻觉出不存在的问题。
- 正则和规则引擎速度快、成本低、结果稳定。

Audit Agent 检查内容：

1. PHI 扫描。
2. 静态加密。
3. 传输加密。
4. RBAC 访问控制。
5. 审计日志。
6. 最小必要原则。
7. 泄露通知就绪。
8. 数据保留策略。

## 10. GraphRAG 知识图谱

### 10.1 为什么需要 GraphRAG

传统 RAG 主要基于文档切片和向量相似度，适合开放问答，但医疗知识更像一张结构化关系网：

```text
症状 -> 疾病 -> ICD-10 -> DRGs
疾病 -> 治疗方案 -> 药物
药物 -> 药物交互 -> 药物
检查结果 -> 临床证据 -> 疾病
```

GraphRAG 用实体和关系建模，比向量检索更适合医学诊断。

| 维度 | 传统 RAG | GraphRAG |
|------|----------|----------|
| 存储形式 | 文档片段向量 | 实体和关系 |
| 检索方式 | 相似度搜索 | 图遍历、多跳查询 |
| 可解释性 | 较弱 | 强，可以给出关系路径 |
| 多跳推理 | 不擅长 | 天然支持 |
| 医疗场景 | 容易召回不精确 | 适合症状、疾病、药物、编码关系 |

### 10.2 项目中的图谱 Schema

```text
(Disease)-[:HAS_SYMPTOM]->(Symptom)
(Disease)-[:TREATED_BY]->(Treatment)
(Treatment)-[:USES_DRUG]->(Drug)
(Drug)-[:INTERACTS_WITH]->(Drug)
(Disease)-[:CODED_AS]->(ICD10Code)
(ICD10Code)-[:GROUPS_TO]->(DRGGroup)
```

面试时可以举例：

> 如果患者有 fever、cough、chest_pain，GraphRAG 会分别查询这些症状关联的疾病，然后按匹配症状数量排序。肺炎可能匹配 3 个症状，所以排名更靠前。LLM 再结合患者检查结果和病史做最终判断。

### 10.3 离线模式和在线模式

项目支持两种模式：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 离线模式 | 使用 Python 字典维护症状、疾病、ICD-10 映射 | 开发、测试、面试演示 |
| 在线模式 | 连接 Neo4j，通过 Cypher 查询真实知识图谱 | 生产环境 |

离线模式的好处是项目可以快速跑通，不依赖复杂图谱数据导入。生产环境可以导入 UMLS、SNOMED-CT、ICD-10、RxNorm、MIMIC-IV 等数据。

### 10.4 U-Retrieval 思路

U-Retrieval 是 Medical-Graph-RAG 中的双向检索策略：

- Top-down：从疾病类别逐步缩小范围。
- Bottom-up：从症状、实验室结果、影像结果等证据向上追溯疾病。
- 最终融合两类候选结果并排序。

这部分可以作为高级扩展点来讲，说明你不仅实现了基础图谱检索，也理解医学知识图谱的生产化方向。

## 11. FHIR API 集成

FHIR 是 HL7 制定的医疗数据交换标准，可以理解为医疗行业的 REST API 数据模型标准。项目使用 FHIR R4，目标是让系统输出能和医院 HIS/EMR、HAPI FHIR Server、Amazon HealthLake、Google Cloud Healthcare API 等系统对接。

### 11.1 项目中的 FHIR 映射

| 内部数据 | FHIR 资源 | 用途 |
|----------|-----------|------|
| `patient_info` | Patient | 患者人口学信息 |
| `diagnosis` | Condition | 诊断和病情 |
| `treatment_plan.medications` | MedicationRequest | 用药计划 |
| 未来扩展 | Observation | 生命体征和检验结果 |
| 未来扩展 | Encounter | 就诊记录 |
| 未来扩展 | Claim | 医保理赔 |

### 11.2 PatientInfo 到 Patient

关键映射：

- `name` -> `Patient.name[0].text`
- `age` -> 反算 `birthDate`
- `gender` -> `Patient.gender`
- `patient_id` -> `Patient.id`
- `allergies` -> 可扩展为 AllergyIntolerance

### 11.3 Diagnosis 到 Condition

关键映射：

- 诊断名称 -> `Condition.code.text`
- ICD-10 编码 -> `Condition.code.coding[0].code`
- 编码系统 -> `http://hl7.org/fhir/sid/icd-10-cm`
- 推理说明 -> `Condition.note`
- 患者引用 -> `Condition.subject.reference`

### 11.4 Medication 到 MedicationRequest

关键映射：

- 药物名称 -> `medicationCodeableConcept.text`
- 通用名 -> `coding.display`
- 剂量 -> `dosageInstruction`
- 给药途径 -> `route`
- 频率 -> `timing`

面试时可以这样说：

> FHIR 的价值是互操作性。项目内部可以用适合业务的结构化 JSON，但对外集成时转换为 Patient、Condition、MedicationRequest 等标准资源，未来就能接入 HAPI FHIR 或云厂商医疗数据平台。

### 11.5 SMART on FHIR

当前版本没有完整实现 SMART on FHIR，但文档设计了基于 OAuth 2.0 的认证授权流程。可以作为生产化扩展讲：

- 注册系统为 SMART App。
- 用户授权后获取 access token。
- 使用 scope 控制资源权限，例如 `patient/Patient.read`、`patient/MedicationRequest.write`。
- 带 token 访问 FHIR Resource Server。

## 12. HIPAA 合规设计

HIPAA 是美国医疗健康数据保护法规，核心目标是保护 PHI。

PHI 可以理解为：

```text
个人身份信息 + 健康信息 = 受保护健康信息
```

例如：

- “John Smith，45岁，诊断为肺炎”是 PHI。
- “45岁男性，诊断为肺炎”通常不是 PHI，因为无法识别个人身份。

### 12.1 Safe Harbor 去标识化

HIPAA Safe Harbor 要求移除或泛化 18 类标识符，包括姓名、地址、日期、电话、邮箱、SSN、病历号、IP 地址、生物标识、面部照片等。

项目当前实现了核心 9 类 PHI 的正则检测：

| PHI 类型 | 示例 |
|----------|------|
| name | John Smith |
| date_of_birth | 1980-01-15 |
| phone | 555-123-4567 |
| email | john@hospital.com |
| ssn | 123-45-6789 |
| mrn | MRN:12345 |
| ip_address | 192.168.1.1 |
| zip_code | 10001 |
| address | 123 Main Street |

### 12.2 脱敏策略

| 策略 | 说明 | 示例 |
|------|------|------|
| Redaction | 用占位符替换 | SSN -> `[SSN_REDACTED]` |
| Pseudonymization | 哈希假名化 | John Smith -> 固定哈希 |
| Generalization | 泛化精度 | 1980-03-15 -> 1980 |

### 12.3 WORM 审计日志

项目设计了不可变审计日志：

- 每次 Pipeline 执行都记录 PHI 扫描、脱敏、合规评估。
- PostgreSQL 中的 `audit_logs` 表作为审计日志存储。
- 生产环境建议撤销应用用户对审计表的 DELETE 和 UPDATE 权限。
- HIPAA 审计日志通常要求至少保留 6 年。

### 12.4 面试推荐回答

如果面试官问“你的系统如何保证 HIPAA 合规”，可以这样回答：

> 我从检测、脱敏、审计和生产加固四个层面做设计。首先，Audit Agent 会扫描 Pipeline 输出中的核心 PHI 模式；其次，发现 PHI 后会自动做替换或哈希脱敏；第三，每次操作都会写入不可变审计日志；最后，生产环境还需要 TLS、静态加密、RBAC、BAA 和泄露响应流程。并且我没有用 LLM 做合规判断，因为合规场景需要确定性和可复现。

## 13. 错误处理、重试和降级

医疗辅助决策系统不能因为一个 Agent 失败就完全崩溃，所以项目采用错误累积和优雅降级。

### 13.1 错误处理策略

| 错误 | 处理方式 |
|------|----------|
| LLM 返回非 JSON | 剥离 Markdown 代码块后解析，失败则记录到 `errors` |
| OpenAI API 超时 | 重试一次，仍失败则对应字段置空 |
| API Key 无效 | 记录认证错误，LLM Agent 失败但 Pipeline 继续 |
| 输入为空 | Intake 返回错误，后续 Agent 根据空字段跳过依赖逻辑 |
| Neo4j 不可用 | 降级到内置知识库 |
| PostgreSQL 不可用 | 审计日志暂时无法持久化，但可返回内存审计结果 |

### 13.2 条件路由重试

Diagnosis Agent 认为信息不足时，可以回到 Intake Agent 补充信息，但最多重试 2 次，避免死循环。

### 13.3 JSON 解析容错

因为 LLM 有时会返回：

````text
```json
{ ... }
```
````

所以 Agent 会先剥离 Markdown fence，再执行 `json.loads`。

面试时可以强调：

> 我把 LLM 当成不完全可靠的外部依赖来处理。所有 LLM 输出都要结构化解析，失败时记录错误并降级，而不是让异常直接打断整个 Pipeline。

## 14. 测试设计

项目测试分层：

| 层级 | 是否需要 LLM | 测试内容 |
|------|--------------|----------|
| 服务层单元测试 | 不需要 | ICD-10 搜索、DDI 检查、PHI 检测、FHIR 转换 |

| Agent 集成测试 | 需要 | 调用真实 LLM，验证 Prompt 和输出结构 |
| Pipeline 端到端测试 | 需要 | 从患者描述到最终 ClinicalState |

内置典型病例可以覆盖：

- 肺炎 -> J18.1
- STEMI 心梗 -> I21.0
- 甲减 -> E03.9
- 心力衰竭 -> I50.9
- 急性阑尾炎 -> K35.80

面试可以说：

> 我会优先把不依赖 LLM 的服务层测扎实，因为这部分是确定性的；LLM Agent 则用 Mock 测结构，用少量集成测试测真实模型表现，避免测试成本过高。

## 15. 三语言实现对比

项目提供 Python、Java、Go 三种实现，体现跨技术栈理解。

| 维度 | Python | Java | Go |
|------|--------|------|----|
| Web 框架 | FastAPI | Spring Boot 3.3 | Gin |
| Agent 编排 | LangGraph StateGraph | 显式循环 Pipeline | for 循环 + Agent interface |
| LLM 调用 | langchain-openai | Spring AI | go-openai |
| 状态模型 | Pydantic | Lombok POJO | Go struct |
| 优势 | AI 生态最完整，LangGraph 原生 | 企业后端生态成熟 | 部署轻量，性能好 |

面试时可以这样讲：

> Python 版本是主实现，适合展示 LangGraph 和 AI 工程；Java 版本展示企业级后端集成 LLM 的思路；Go 版本展示如果不用框架，Agent Pipeline 底层其实就是状态对象加接口加条件循环。

## 16. 部署和运维设计

### 16.1 部署模式

| 模式 | 场景 |
|------|------|
| Docker Compose | 开发、测试、小规模部署 |
| Kubernetes | 中大规模生产、自动扩缩容、滚动更新 |
| 云托管 | 快速上线，减少基础设施维护 |

本地依赖：

- PostgreSQL：审计日志和会话。
- Neo4j：知识图谱。
- Redis：缓存和异步任务。
- OpenAI API：LLM 调用。

### 16.2 Kubernetes 生产化要点

- API Deployment 设置多副本。
- Service 做服务发现。
- Ingress 做 HTTPS 路由和 TLS 终止。
- Secret 管理 OpenAI Key 和数据库密码。
- HPA 根据 CPU/内存自动扩缩容。
- PostgreSQL 用 StatefulSet + PVC，生产更推荐云托管 RDS。

### 16.3 监控告警

关键指标：

| 类型 | 指标 | 告警示例 |
|------|------|----------|
| 延迟 | API P99、Pipeline P95 | P99 > 5s 或 Pipeline > 60s |
| 错误 | 5xx、LLM 调用失败率 | 5xx > 1%，LLM 错误 > 5% |
| LLM | `llm_call_duration_seconds` | P99 > 30s |
| 数据库 | PostgreSQL up、连接数 | DB down 或连接数过高 |
| 资源 | CPU、内存 | 使用率 > 80% |

方案：

- Prometheus 采集指标。
- Grafana 可视化。
- AlertManager 告警。
- ELK 收集结构化日志。

### 16.4 日志注意事项

生产环境日志不能包含 PHI。

建议：

- 开发环境 DEBUG 可以临时记录 LLM 输入输出。
- 测试环境 INFO 记录关键流程。
- 生产环境 WARNING，只记录异常和审计事件。
- PHI 相关操作只进入受控审计日志。

### 16.5 性能瓶颈和优化

项目典型耗时：

| Agent | 典型耗时 |
|-------|----------|
| Intake | 2-5s |
| Diagnosis | 3-8s |
| Treatment | 3-8s |
| Coding | 2-5s |
| Audit | 约 10ms |

总耗时约 10-26s，瓶颈主要是 LLM API。

优化策略：

1. 使用更快模型，例如 gpt-4o-mini。
2. Redis 缓存相同或相似输入的 LLM 结果。
3. 异步处理，API 返回 task_id，后台执行 Pipeline。
4. Streaming 输出，降低用户感知延迟。
5. 对无依赖 Agent 做并行化。
6. 数据库和 HTTP 使用连接池。

### 16.6 CI/CD 和安全加固

CI/CD 流程：

```text
代码提交 -> 测试 -> 构建镜像 -> 安全扫描 -> 推送镜像 -> 部署 Staging -> 冒烟测试 -> 生产滚动发布
```

安全加固：

- 全链路 TLS。
- 数据库不暴露公网。
- API Key 不硬编码。
- 使用 Vault / KMS / Kubernetes Secret 管理密钥。
- 容器使用非 root 用户运行。
- 镜像漏洞扫描。
- 配置 CORS 和速率限制。
- 备份采用 3-2-1 原则。

## 17. 项目亮点总结

### 亮点一：多 Agent 拆分复杂医疗流程

不是用一个大 Prompt 完成所有事，而是把临床决策拆成五个专业 Agent。每个 Agent 职责单一、输入输出清晰、可独立测试。

### 亮点二：Pipeline 编排符合医疗合规场景

医疗流程要求确定性、可追溯和成本可控，因此选择 Pipeline。通过 `needs_more_info` 条件路由保留一定灵活性。

### 亮点三：GraphRAG 降低 LLM 幻觉

Diagnosis Agent 不完全依赖 LLM，而是先从症状到疾病的知识图谱中召回候选，再让 LLM 基于候选和患者信息推理。

### 亮点四：合规检查不用 LLM

Audit Agent 使用正则和规则引擎做 PHI 检测、脱敏和审计。这个设计体现了对 LLM 适用边界的理解。

### 亮点五：对齐医疗行业标准

输出支持 FHIR R4 资源映射，诊断支持 ICD-10 编码和 DRGs 分组，隐私保护对齐 HIPAA。

### 亮点六：工程化完整

项目覆盖 Docker、K8s、监控告警、结构化日志、错误降级、异步处理、缓存、CI/CD 和安全加固，不只是 Demo。

## 18. 项目局限性和改进方向

面试中主动讲局限性会显得更真实。

| 当前局限 | 改进方向 |
|----------|----------|

## 19. 高频面试问答

### Q1：为什么不用单 Agent？

单 Agent 会导致 Prompt 很长、输出巨大、难以定位错误，也很难独立测试。多 Agent 把任务拆成多个专业环节，每个 Prompt 更短，职责更清晰，出错时能定位到具体 Agent，也方便扩展新能力。

### Q2：为什么不用 Supervisor？

Supervisor 适合任务类型不确定的场景，但医疗临床辅助决策流程本身很固定，而且需要可审计。Pipeline 的执行路径更确定、成本更稳定，更适合合规场景。

### Q3：GraphRAG 和传统 RAG 有什么区别？

传统 RAG 基于文档片段和向量相似度，GraphRAG 基于实体和关系。医疗知识天然是“症状、疾病、药物、编码、治疗方案”的关系网络，所以图谱能支持更精确的多跳推理，也更容易解释推理路径。

### Q4：你怎么控制 LLM 幻觉？

主要有四层控制：第一，Prompt 要求结构化 JSON 输出；第二，GraphRAG 提供候选疾病约束诊断；第三，ICD-10、DDI、HIPAA 等确定性逻辑放在服务层或规则引擎中；第四，解析失败时记录错误并降级，而不是直接信任模型。

### Q5：为什么 Audit Agent 不使用 LLM？

合规检查要求确定性、可复现、可审计。LLM 可能漏检或误报 PHI，也有成本和延迟。正则和规则引擎更适合 PHI 检测、脱敏和审计日志生成。

### Q6：如何保证系统可扩展？

新增 Agent 只需要三步：增加 Agent 函数、在 ClinicalState 加字段、在 Pipeline 中注册节点和边。服务层能力也独立封装，比如 FHIR、HIPAA、GraphRAG、ICD-10 都能单独替换或增强。

### Q7：系统瓶颈在哪里？

瓶颈主要是 LLM API 调用，Audit Agent 这类规则环节只有毫秒级。优化可以从更快模型、缓存、异步任务、流式输出和并行化入手。

### Q8：生产部署需要补什么？

需要补完整认证授权、RBAC、TLS、静态加密、密钥管理、审计日志持久化、备份恢复、告警、灰度发布、BAA、第三方安全审计和医生审核流程。

### Q9：FHIR 在项目中有什么价值？

FHIR 让系统输出能和医疗生态互通。内部 Pipeline 生成的是项目自己的结构化数据，对外可以转换成 Patient、Condition、MedicationRequest 和 Bundle，方便接入 HAPI FHIR、医院 HIS/EMR 或云医疗平台。

### Q10：如果 OpenAI API 不可用怎么办？

系统可以降级：不依赖 LLM 的 ICD-10 搜索、DDI 检查、PHI 检测仍可运行；LLM 相关 Agent 记录错误并返回部分结果。生产环境还可以配置备用模型、指数退避重试和异步任务队列。
